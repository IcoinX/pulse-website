# F1: Notifications Server-Side — Implementation Guide

## Architecture Choices (Validated)

| Component | Choice | Rationale |
|-----------|--------|-----------|
| **Channel** | Telegram Bot | Crypto-native adoption, instant push, simple UX |
| **Queue** | `notifications` table with status | Single table for queue + audit, sufficient for <100k notifs/day |
| **Idempotence** | UUID v5 | Deterministic, collision-free, debuggable |
| **Worker** | Cron 30s + Node.js | Predictable, resilient, no real-time magic while protocol is young |

## Database Schema

### Tables

#### `notification_preferences` — Opt-in Registry
```sql
user_id TEXT PRIMARY KEY           -- Wallet or user identifier
event_status_change BOOLEAN        -- FALSE by default (opt-in required)
challenge_created BOOLEAN          -- FALSE by default
challenge_resolved BOOLEAN         -- FALSE by default
channel TEXT                       -- 'telegram' | 'discord' | 'email'
channel_config JSONB               -- {telegram_chat_id: "..."} etc.
max_per_day INTEGER DEFAULT 10     -- Rate limit per user
notifs_sent_today INTEGER          -- Daily counter
day_reset_at TIMESTAMPTZ           -- Last reset timestamp
```

#### `notifications` — Queue + Audit Log
```sql
id UUID PRIMARY KEY
type TEXT                          -- EVENT_STATUS_CHANGE | CHALLENGE_CREATED | CHALLENGE_RESOLVED
entity_type TEXT                   -- 'event' | 'challenge'
entity_id BIGINT                   -- event_id or challenge_id
user_id TEXT                       -- Target user
title TEXT, body TEXT, url TEXT    -- Content snapshot
channel TEXT                       -- Delivery channel
status TEXT                        -- pending | sending | sent | failed | bounced
idempotency_key UUID UNIQUE        -- UUID v5(type:entity:user)
attempts INTEGER                   -- Retry counter
next_retry_at TIMESTAMPTZ          -- Scheduled retry time
telegram_message_id TEXT           -- For potential edits/deletion
```

### Triggers (3 Strict Whitelist)

1. **EVENT_STATUS_CHANGE**
   - Fires on: `events.verification_status` update
   - Condition: PENDING/DISPUTED → VERIFIED/REJECTED
   - Rate limit: Respects user max_per_day

2. **CHALLENGE_CREATED**
   - Fires on: `challenges` INSERT
   - Condition: `challenger_stake >= 50`
   - Rate limit: Respects user max_per_day

3. **CHALLENGE_RESOLVED**
   - Fires on: `challenges.status` → 'RESOLVED'
   - Condition: Always (if user opted in)
   - Rate limit: Respects user max_per_day

## Retry Strategy

| Attempt | Delay | Status |
|---------|-------|--------|
| 1 | 0s | Try immediately |
| 2 | +30s | Retry |
| 3 | +2min | Retry |
| 4 | +5min | Retry |
| 5 | +10min | Retry |
| 6+ | — | Mark as failed |

Failed notifications are logged but not retried indefinitely.

## Idempotence

UUID v5 generated from:
```
namespace: nil UUID
data: "{TYPE}:{entity_type}:{entity_id}:{user_id}"

Example: "EVENT_STATUS_CHANGE:event:12345:0xAbC..."
```

This ensures identical notifications are never sent twice, even on trigger re-execution.

## Rate Limiting

Per-user daily limits enforced:
- Default: 10 notifications/day
- Configurable per user via `max_per_day`
- Resets automatically every 24h

1. **Token Storage**: `TELEGRAM_BOT_TOKEN` in `.env.local`, never committed
2. **Webhook Secret**: Verify `X-Telegram-Bot-Api-Secret-Token` header
3. **Rate Limiting**: Prevents abuse of notification system
4. **Opt-in Only**: No notifications without explicit consent
5. **Audit Trail**: Complete history in `notifications` table

## Troubleshooting

### Bot not responding to /start
- Check webhook URL is HTTPS and accessible
- Verify `TELEGRAM_BOT_TOKEN` is correct
- Check `notification_cron.sh` logs

### Notifications not sending
- Verify `notification_preferences` has opted-in user
- Check `notifications` table for pending items
- Run worker manually: `node scripts/notification_worker.js`
- Check Supabase RLS policies (should allow service key access)

### Duplicate notifications
- Verify `idempotency_key` is being generated correctly
- Check for multiple triggers firing
